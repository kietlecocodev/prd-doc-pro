# Bad Example: Problem Statement (Mobile Fintech)

**Score: 2/10** — Vague, solution-focused, no data

---

## Problem Statement

We need to improve the payment transfer experience in our mobile app. Many users have reported issues with the current flow and it could be better. The current transfer process is too complicated and needs to be simplified. We should add a new transfer feature that makes it easier and faster.

---

## What's Wrong (Annotated)

| Antipattern | Where It Appears |
|-------------|-----------------|
| **Solution-as-Problem** | "We should add a new transfer feature" — this is a solution, not a problem |
| **Unquantified** | "Many users" — how many? "Issues" — what issues? "Too complicated" — compared to what? |
| **No personas** | "Users" — which users? New users? Power users? All equally affected? |
| **No business impact** | No mention of revenue, churn, support cost, or competitive pressure |
| **No evidence** | No data source cited. No user research. No analytics. |
| **Prescriptive** | "Simplified" and "easier and faster" prescribe outcomes without defining the actual pain |

## How to Fix It

Rewrite by answering these 4 questions:
1. **Who** is affected? (specific persona with context)
2. **What** do they suffer? (observable behavior, not opinion)
3. **How bad** is it? (numbers: %, counts, dollars, time)
4. **Why should we care?** (business impact: revenue, churn, cost)

Then remove ALL mentions of solutions, features, or what to build.
